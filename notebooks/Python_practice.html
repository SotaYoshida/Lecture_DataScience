練習帳¶

このノートブックでは、1-8章で抑えておくべき基本事項を確認するための練習問題を与える。

CLで収集する授業のリアクションシート(第3回10/24-第9回12/5)には、

  • 練習帳のURL

  • 授業や資料でわからなかったこと

  • なければ適当な感想(最近ハマっていることやお昼ごはんなど)

を当該の週の水曜17:00までに提出すること。これをもって出席とみなす。

この練習帳の目的は、皆さんが自分自身の理解・疑問点を確認するためであって 練習帳で書いているコードなどが正しいかどうかは一切授業の評点には関係がない。 したがって友人に見せてもらったり相談をして体裁を整える必要はまったくないので 教員やTAへの質問を円滑にするための連絡帳のようなものだと思ってもらえれば良い。

また、質問の見落としを防ぐためリアクションシート(CL)の方で

練習帳に詳細な質問とコードを書きました

などと教えてもらえるとスムーズな対応が可能になります。

練習帳のノートブックの共有の仕方¶

以下の手順にならってください。 1-3は1度やれば再び行う必要はありません。

  1. まずノートブックのコピーを作成し、ファイル名を適当に編集する: 例: 練習帳_123456X.ipynb (123456Xは学籍番号のつもり)

  2. 右上の共有ボタンを押し、「制限付き」を「リンクを知っている全員」に変更 (共有ボタンが見えない場合、編集権限がないつまり「ノートブックのコピーをつくる」というお約束を忘れていることを意味する)

  3. 「リンクを知っている全員」の右にある「閲覧者」を「編集者」に変更する

  4. 最後に、リンクをコピーし、どこかにメモっておこう(2回目以降はCLで一度提出した過去のURLをコピーするのが手っ取り早い)

最終課題¶

授業の第12-15回で取り組む最終課題については、自由な発想で課題を設計し、取り組んでもらって構わない。 一方で「教員やTAに相談したくない」「レポートの得点が比較的低かった」という方には自由課題は正直オススメできない。

自由課題に困っている方に向けて予めテンプレ的な課題を3つほど用意してある。自由課題/テンプレ課題の選択で得点が異なることはない。
「これをやれ!!」という訳ではもちろんなく、課題が思いつかない方への一種の救済措置のようなものだと捉えてほしい。
以下の例はあくまで必要な情報の例で、加えて分析・考察・機能などを追加すると、もちろん加点対象となる。

テンプレ課題1: オープンデータの収集と可視化¶

政府統計などのオープンデータをなるべく多く取得して、可視化したり考察をしてみよう。

  1. まずは、ほしいデータの種類と収集方法を考えよう。政府統計ならe-Stat,他にも各地方自治体などがデータを提供していることも多い。

  2. データの種類と収集先の目処が経ったら、まずは利用規約を読んでスクレイピング等を禁止していないかを調べる。

  3. 問題がないことを確認したら、データの収集にあたる。APIなどのデータ取得用の窓口があれば必ずそちらを利用する。

  4. 次に得られたデータを時系列や他のデータとの相関、年代ごとの棒グラフなど様々な方法で可視化する。

  5. 可視化されることで得られる理解や予想、その背景について考察を行う。

テンプレ課題2: 機械学習を用いた画像認識/画像生成¶

  1. 画像認識や画像生成を行う機械学習モデルについて概観し、授業を受講している他の学生に説明できるよう解説する。 その際、情報の参照元は論文が望ましいが、難しければ日本語等の解説を参照して構わない。(注:もちろんレポート内では適切に引用・参照すること)

  2. 次に、それらの機械学習モデルの中から、PythonがインターフェースとなっていてGoogleColab上で容易に実行ができそうなものを選ぶ。例えば…

    • YOLOv5: CNNを用いた画像認識のパッケージ

    • StableDiffusion: 潜在拡散モデルを用いた画像生成モデル

  3. これらを実際に動かして画像認識や画像生成を実行してみる。

  4. 機械学習モデルによる画像認識/画像生成の技術について、サービスや企業での導入事例などを調べると同時に、こうした技術の使用に際して起こり得るトラブルや問題点について考察する。

テンプレ課題3: 簡単なゲーム開発¶

数あてゲームやブラックジャックやババ抜きといったトランプを用いたゲーム、オセロなどを作る。 (Pythonにはゲーム作成のためのライブラリなどもあるが、これらは自由課題にあたる)

  1. そのために、まずはターンや盤面の状態を表現するための方法を考えよう。例えば…

    • 山札/場、人数分のリストを用意しておいて手札や盤面などを表現する

    • ターンが進行するたびに絵をplotする仕組みにして、視覚的に表現する

    など、考え方は色々ある。(参考:コードの実行中に適宜入力を受け取るようなことももちろんできる。)

  2. ゲーム開始の状況を表現するための方法を考える(山札をランダムに配る, 駒の初期配置など)

  3. 比較的簡単な状況(参加人数はできるだけ少なくルールも単純化する)でプログラムを作ってみる。例えばオセロだと4×4にする。

  4. 特殊な場合(ババ抜きで配った瞬間誰かがあがってしまう, パスが必要)に対する対策を講じる。

  5. 他人に遊んでもらえるようルールを説明するテキスト部分を用意する。(オセロの例なら、どうやって置く駒の座標を指定するか。キーボードで操作するのかクリックなのかなど。)
